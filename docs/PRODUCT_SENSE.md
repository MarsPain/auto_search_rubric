> 产品愿景、用户场景、核心价值的统一理解。

---

## 产品愿景

**让高质量的评估标准（Rubric）成为可工程化、可自动化、可闭环迭代的生产要素。**

Reward Harness（历史名称 AutoSR）不是简单的"搜索更好的prompt"，而是构建一个从搜索到部署、从训练到蒸馏、再到监控的完整奖励模型（Reward Model）工程体系。

---

## 核心用户场景

### 场景1: 搜索最优Rubric

**用户**: 算法工程师
**问题**: 如何为特定任务找到最优的评估标准？
**解决方案**: 
- 输入：示例数据集、评估维度
- 输出：经过进化/迭代优化的Rubric
- 价值：自动化、系统性地探索标准空间

### 场景2: 可复现的RM部署

**用户**: MLEngineer / 平台工程师
**问题**: 如何将搜索到的Rubric稳定地部署为线上服务？
**解决方案**:
- RMArtifact版本化管理
- 可追溯的发布记录
- hash一致性校验

### 场景3: RL训练奖励信号

**用户**: RL研究员
**问题**: 如何获得稳定、可比较的奖励信号进行训练？
**解决方案**:
- RM Server提供标准化评分API
- 训练实验与RM版本绑定
- 效果可追溯、可对比

### 场景4: 基于RL采样蒸馏Classifier RM

**用户**: RL研究员 / RM工程师
**问题**: 如何把高成本 rubric RM 在 RL 采样分布上蒸馏成更便宜的 classifier RM？
**解决方案**:
- 自动收集 RL 训练中的 `(query, response)` 采样
- 多次重复打分降噪
- 自动构造偏好数据并触发外部 trainer

### 场景5: 效果监控与迭代

**用户**: 产品/算法负责人
**问题**: 如何知道当前RM是否仍然有效？何时需要刷新？
**解决方案**:
- 监控训练/评测指标
- 自动告警与回归检测
- 闭环触发重新搜索

## 核心价值主张

| 价值 | 说明 | 与传统方式对比 |
|------|------|----------------|
| **工程化** | Rubric从"手工撰写"变为"可编程、可自动化" | 不再是文档，而是代码 |
| **可追溯** | 每个Rubric都可追溯到搜索会话、数据集版本 | 不再是谁写的忘记了 |
| **可比较** | 不同Rubric的效果可以科学对比 | 不再是"感觉这个更好" |
| **可回滚** | 效果退化时可以快速回滚 | 不再是出了问题只能重新来 |
| **可闭环** | 从评测结果自动触发优化 | 不再是人工发现、人工触发 |

## 设计原则

### 1. 闭环优先

任何新增能力必须服务 `Search -> RM -> RL -> Classifier RM -> Eval` 链路。

**反例**: 一个只在本地好用但无法导出为artifact的功能。
**正例**: 一个支持版本对比的artifact浏览器。

### 2. 契约先行

先定义artifact/API/指标契约，再做实现。

**反例**: 先写代码，再补文档说明输入输出。
**正例**: 先定义RMArtifact schema，再实现导出功能。

### 3. 可回滚

每次RM发布、每次训练实验都可追溯到明确版本。

**反例**: 线上RM升级后发现有问题，但不知道上一个版本是什么。
**正例**: 发布自带manifest，升级失败一键回滚。

### 4. 兼容现有CLI

`uv run python -m autosr.cli` 继续可用。

**反例**: 新版本完全废弃旧命令格式。
**正例**: 新能力通过新增命令或可选参数引入。

### 5. 先单机后分布式

未证明价值前不提前引入复杂基础设施。

**反例**: 一上来就设计复杂的分布式架构。
**正例**: 单机闭环先跑通，再评估扩展需求。

## 北极星指标

1. **搜索效率**: 从启动到获得可用Rubric的时间
2. **部署成功率**: RM artifact部署成功的比例
3. **训练稳定性**: RL训练因RM问题中断的频率
4. **效果可追溯性**: 从训练结果反查RM版本的平均时间
5. **闭环延迟**: 从检测到退化到完成新Rubric搜索的时间

---

## 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| RM与离线评分不一致 | 高 | 建立一致性测试集，发布前强制比对 |
| 训练结果不可追溯 | 高 | 强制run manifest记录artifact/data/code版本 |
| Reward hacking | 高 | 引入独立holdout评测与多指标约束 |
| RM服务抖动 | 高 | 设定超时/重试/降级策略 |
| 过早追求分布式 | 中高 | 单机闭环先跑通 |

## 产品演进路线

### 当前阶段：工程底座 ✅（历史名称 AutoSR）

- 搜索可恢复、可复现
- Artifact可导出、可校验

### 下一阶段：服务化与训练接入 🚧

- RM Server可部署
- 训练可接入

### 中期阶段：蒸馏化 📋

- 基于RL采样自动训练classifier RM
- 偏好数据与teacher RM可追溯

### 远期阶段：智能化 📋

- 自动监控与告警
- 退化自动触发重搜
- canary发布与回滚
